# KCC Direct Migration Testing

The Migration Test Suite (`TestMigrationToDirect` in [e2e/migration_test.go](file:///usr/local/google/home/anhdle/go/src/github.com/anhdle-sso/k8s-config-connector/tests/e2e/migration_test.go)) verifies the safe transition of Kubernetes Custom Resources in Config Connector (KCC) from legacy controllers (Terraform-based or DCL) to the new **Direct** controllers (built on Google Cloud SDKs and `controller-runtime`).

Because KCC manages production infrastructure at scale, migrating an existing resource type to a Direct controller must guarantee **100% backward compatibility** and **zero unexpected mutations**. The same KCC YAML must continue to produce and manage the exact same GCP resources without downtime, drift, or unintended API write traffic.

---

## The 4-Phase Migration Lifecycle

`TestMigrationToDirect` executes a structured 4-phase test against test fixtures in `pkg/test/resourcefixture/testdata/basic/`:

```
+-----------------------------------------------------------------------------------+
| Phase 1: Legacy Provisioning (Create)                                             |
| - Controller: Legacy (Terraform / DCL)                                            |
| - Action: Create primary resource + dependencies from create.yaml                 |
| - Output: _http_migration_phase1.log                                              |
+-----------------------------------------------------------------------------------+
                                         |
                                         v
+-----------------------------------------------------------------------------------+
| Phase 2: Legacy Re-reconciliation (No-op Update)                                  |
| - Controller: Legacy (Terraform / DCL)                                            |
| - Action: Apply no-op annotation (re-reconcile-legacy-v1)                         |
| - Output: _http_migration_phase2.log                                              |
+-----------------------------------------------------------------------------------+
                                         |
                                         v
+-----------------------------------------------------------------------------------+
| Phase 3: Direct Controller Takeover                                               |
| - Controller: Direct                                                              |
| - Action: Apply 'reconciler: direct' annotation                                   |
| - Requirement: MUST be a clean 0-write no-op reconciliation                       |
| - Output: _http_migration_phase3.log                                              |
+-----------------------------------------------------------------------------------+
                                         |
                                         v
+-----------------------------------------------------------------------------------+
| Phase 4: Direct Controller Re-reconciliation                                      |
| - Controller: Direct                                                              |
| - Action: Apply no-op annotation (re-reconcile-direct-v1)                         |
| - Requirement: MUST remain stable with 0 write requests                           |
| - Output: _http_migration_phase4.log & _migration_diffs.json                      |
+-----------------------------------------------------------------------------------+
```

### Phase 1: Legacy Provisioning (Create)
The resource and its dependencies (from `create.yaml` and `dependencies.yaml`) are provisioned in GCP using the legacy controller. The harness waits for all resources to report `status.condition` Ready (`UpToDate`). All HTTP and gRPC traffic generated by the legacy controller is recorded to `_http_migration_phase1.log`.

### Phase 2: Legacy Re-reconciliation (No-op Update)
A no-op annotation (`test.cnrm.cloud.google.com/reconcile-cookie: re-reconcile-legacy-v1`) is applied to the resource while the legacy controller is still active. This tests the baseline stability of the legacy controller during re-reconciles. Any API traffic generated is recorded to `_http_migration_phase2.log`.

### Phase 3: Direct Controller Takeover
The resource is patched with the annotations `alpha.cnrm.cloud.google.com/reconciler: direct` and `test.cnrm.cloud.google.com/reconcile-cookie: migration-v1`. (The reconcile cookie is required because KCC's event filter predicate in `pkg/controller/predicate/predicate.go` ignores changes to the reconciler annotation alone). This instructs KCC to dynamically switch routing from the legacy controller to the Direct controller.
* **Critical Requirement**: When the Direct controller adopts an already-provisioned resource, it must perform a **0-write no-op reconciliation**. The controller should only issue read requests (`GET` / `List`).

* If the Direct controller issues any write request (`POST`, `PUT`, `PATCH`, `DELETE`) during takeover, the test will fail with:
  `FAIL: unexpected write request during migration reconciliation: <METHOD> <URL>`.
* Read-only verification traffic is recorded to `_http_migration_phase3.log`.

### Phase 4: Direct Controller Re-reconciliation
Another no-op annotation (`test.cnrm.cloud.google.com/reconcile-cookie: re-reconcile-direct-v1`) is applied while the resource is managed by the Direct controller.
* **Critical Requirement**: The Direct controller must verify that the desired state matches the live GCP state without issuing any modifications.
* Traffic is recorded to `_http_migration_phase4.log`, and any structured diffs detected during the test are serialized to `_migration_diffs.json`.

## Running Migration Tests

### 1. Recording Golden Files (Real GCP)
When qualifying a Direct controller or updating fixtures, run the migration suite against real GCP with golden output recording enabled:

```bash
E2E_KUBE_TARGET=envtest E2E_GCP_TARGET=real GOLDEN_REQUEST_CHECKS=1 GOLDEN_OBJECT_CHECKS=1 WRITE_GOLDEN_OUTPUT=1 RUN_E2E=1 go test ./tests/e2e -v -run TestMigrationToDirect/fixtures/<fixture-name>$
```

*Replace `<fixture-name>` with the name of the test fixture (e.g., `computenetworkdeleteoncreate`).*

### 2. Running Hermetic Tests (MockGCP)
To verify that the migration works hermetically against MockGCP (used in CI/presubmits):

```bash
E2E_KUBE_TARGET=envtest E2E_GCP_TARGET=mock GOLDEN_REQUEST_CHECKS=1 GOLDEN_OBJECT_CHECKS=1 RUN_E2E=1 go test ./tests/e2e -v -run TestMigrationToDirect/fixtures/<fixture-name>$
```

### 3. Filtering Test Execution
You can use environment variables to filter which resources or API groups are tested:
* `ONLY_TEST_APIGROUPS`: Comma-separated list of API groups to test (e.g., `ONLY_TEST_APIGROUPS=compute.cnrm.cloud.google.com`).
* `SKIP_TEST_APIGROUP`: Comma-separated list of API groups to skip.
* `SKIP_ALL`: Skip the entire migration test suite if set.
* `USE_FULL_TEST_NAMES`: When set to `true`, test names use full file paths (e.g., `pkg/test/resourcefixture/testdata/basic/...`).

## Diagnosing & Fixing Migration Failures

When a migration test fails, it is almost always due to a **takeover diff** in Phase 3 or Phase 4, causing the Direct controller to issue unexpected write requests.

### Step 1: Inspect `_migration_diffs.json`
Open the generated `_migration_diffs.json` file in the fixture directory. Look for entries where `"isNewObject": false`.
* Entries during Phase 3 or Phase 4 indicate that the Direct controller perceived a drift between the Kubernetes spec and the live GCP resource.
* Example diff entry indicating a mismatch:
  ```json
  {
    "isNewObject": false,
    "resource": "",
    "fields": [
      {
        "id": "network_firewall_policy_enforcement_order",
        "old": "AFTER_CLASSIC_FIREWALL",
        "new": ""
      }
    ]
  }
  ```

### Step 2: Identify the Root Cause
1. **Server-Side Defaults**: GCP often populates default fields on created resources (e.g., `networkFirewallPolicyEnforcementOrder: "AFTER_CLASSIC_FIREWALL"` or `activationPolicy: "ALWAYS"`). If the Kubernetes spec omits these optional fields (desired is `nil`), the controller will see a diff unless it is programmed to ignore or preserve them.
2. **Perpetual Diffs (Controller Flapping)**: If the controller sees a diff for an omitted field and issues a `PATCH`, but the `PATCH` payload omits the field, GCP leaves the default unchanged. On the next reconcile, the controller sees the exact same diff again, causing an infinite loop of `GET` -> `PATCH` -> `GET` cycles.
3. **Derived / Computed Fields**: Server-generated metadata, schemas, or timestamps returned by GCP must not be compared against desired specs.
4. **Casing & Formatting**: GCP API responses may differ in casing (`INT64` vs `integer`) or URL formatting (relative vs. absolute self-links).

### Step 3: Implement Two-Stage Defaulting
In your Direct controller (`pkg/controller/direct/<service>/<resource>_controller.go`), ensure your comparison logic (`compare[Resource]`) and defaulting handlers follow the **Two-Stage Defaulting Pattern**:
* **Stage 1 (Client-Side Defaults)**: Apply standard client-side defaults for new resources.
* **Stage 2 (Preserve Unmanaged Fields)**: If an optional field is not specified in the user's Kubernetes spec (`desired == nil`), preserve the live value from GCP (`actual`) rather than attempting to clear it.

Example pattern in comparison logic:
```go
func compareResource(ctx context.Context, actual, desired *pb.Resource) (*structuredreporting.Diff, error) {
    maskedActual, err := mappers.OnlySpecFields(actual, Spec_FromProto, Spec_ToProto)
    if err != nil {
        return nil, err
    }
    clonedDesired := proto.CloneOf(desired)

    // Preserve actual server-side defaults when the desired spec is unspecified (nil)
    if clonedDesired.SomeOptionalField == nil && maskedActual.SomeOptionalField != nil {
        clonedDesired.SomeOptionalField = maskedActual.SomeOptionalField
    }

    diffs, _, err := tags.DiffForTopLevelFields(ctx, clonedDesired.ProtoReflect(), maskedActual.ProtoReflect())
    return diffs, err
}
```

### Step 4: Validate and Clean Up
1. Re-run the migration test with `WRITE_GOLDEN_OUTPUT=1` and verify that `_migration_diffs.json` contains no `"isNewObject": false` blocks from Phase 3 or Phase 4.
2. Verify that `_http_migration_phase3.log` and `_http_migration_phase4.log` contain **only read-only requests** (`GET` / `List`).
3. Run `make fmt` and clean up any untracked log files:
   ```bash
   rm -f pkg/test/resourcefixture/testdata/basic/*/*/*/*/_http_migration_phase*.log
   rm -f pkg/test/resourcefixture/testdata/basic/*/*/*/*/_migration_diffs.json
   ```
