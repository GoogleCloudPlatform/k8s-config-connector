### [2026-06-13] Multi-Service Type Generation in Monitoring Group
- **Context**: Implementing code generation for `MonitoringDashboard` under the `monitoring.cnrm.cloud.google.com/v1beta1` group, which is in a different proto package (`google.monitoring.dashboard.v1`) from existing resources (`google.monitoring.v3`).
- **Problem**: When a single API group/version directory hosts resources belonging to different GCP proto services, running `generate-types` sequentially can result in later commands overwriting `types.generated.go` which was created by earlier commands.
- **Solution**: Order the type generation commands in `generate.sh` so that the custom-packaged service (`google.monitoring.dashboard.v1`) is generated first, and immediately rename the output file to `<resource_name>_types.generated.go` (e.g., `dashboard_types.generated.go`). Run the main service type generator (`google.monitoring.v3`) afterwards to keep its output in the standard `types.generated.go`. Add all relevant proto services to the `generate-mapper` command to generate all mappers correctly.
- **Impact**: Prevents unintentional deletion/loss of generated types when multiple proto services map to the same KCC API directory.
