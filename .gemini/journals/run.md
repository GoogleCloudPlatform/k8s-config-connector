### [2026-05-27] Run WorkerPool Proto Field Omission
- **Context**: Implementing the `CloudRunWorkerPool` resource KRM types.
- **Problem**: The KRM `Spec` generated by `controllerbuilder` was nearly empty and omitted a majority of the `WorkerPool` fields (e.g. `Template`, `Scaling`, `Description`, `InstanceSplits`, etc.) and the `ObservedState` was empty because `controllerbuilder` commented out types not referenced directly from the initial minimal `Spec`/`ObservedState` it scaffolded.
- **Solution**: Manually populated the missing fields into `CloudRunWorkerPoolSpec` and `CloudRunWorkerPoolObservedState` in `workerpool_types.go` based on the `.build/third_party/googleapis/google/cloud/run/v2/worker_pool.proto` definition. Using `*` pointers for scalar primitives. Re-running `generate.sh` successfully un-commented the associated structural types in `types.generated.go`.
- **Impact**: When scaffolding new resources, if the root `Spec` or `ObservedState` appears empty, the generated code might have hidden the nested types in `types.generated.go`. Agents must manually map the desired fields from the proto to the `Spec` and `ObservedState` to link to the structures inside `types.generated.go`.
