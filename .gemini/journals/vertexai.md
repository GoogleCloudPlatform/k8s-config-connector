### [2026-05-28] Vertex AI Types Overwriting Conflict in Code Generation
- **Context**: Greenfield direct controller resource types implementation for VertexAINasJob.
- **Problem**: In `apis/vertexai/v1alpha1/generate.sh`, calling `generate-types` multiple times with different proto services (e.g., `google.cloud.aiplatform.v1` and `google.cloud.aiplatform.v1beta1`) causes the subsequent execution to overwrite the entire `types.generated.go` file. This deletes the types generated by the previous command, causing other resource types under the same package (such as `Featurestore`, `MetadataStore`, `DeploymentResourcePool`, `ExampleStore`) to fail compiling because their referenced generated types got pruned/commented out.
- **Solution**: Combine all resources into a single `generate-types` invocation under the more comprehensive `google.cloud.aiplatform.v1beta1` proto service, which contains both `v1` and `v1beta1` types for all vertexai KCC resources in `v1alpha1`.
- **Impact**: Avoids split-generation file-overwriting issues, keeps `types.generated.go` complete and correct, and ensures all vertexai resources under KCC compile successfully.
