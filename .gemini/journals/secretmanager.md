### [2026-05-12] SecretManagerRegionalSecret Multi-version Mapper Quirks
- **Context**: Implementing direct types for `SecretManagerRegionalSecret` (v1alpha1) alongside existing `SecretManagerSecret` (v1beta1).
- **Problem**: Both KRM types use the same proto message `google.cloud.secretmanager.v1.Secret`. When running the `generate-mapper` tool using `--service google.cloud.secretmanager.v1`, the tool overwrites the existing mappings in `pkg/controller/direct/secretmanager/mapper.generated.go` because it expects a 1:1 mapping between a proto service and a single KRM version directory. Since `RegionalSecret` lacks `Replication` in its spec, the generated mapper drops the `Replication` mapping, breaking the `v1beta1` `SecretManagerSecret` compilation.
- **Solution**: Do not rely on `generate-mapper` to regenerate the full `mapper.generated.go` for multi-version setups. Instead, I manually copied the generated mapping logic into a separate file `pkg/controller/direct/secretmanager/regionalsecret_mapping.go`, which implements the mapping exclusively for the `v1alpha1` Regional Secret types.
- **Impact**: The next agent should be cautious when running `generate-mapper` for services that have multiple KRM types mapping to the exact same proto message, as it may silently overwrite existing critical mapping logic. Always review the diffs in `mapper.generated.go` after running code generation.
