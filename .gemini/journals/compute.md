### 2026-06-13 ComputeNetworkEndpointGroup KRM Types Implementation
- **Context**: Implementing direct KRM types and generate.sh configuration for `ComputeNetworkEndpointGroup` under `apis/compute/v1beta1/`.
- **Problem**: The automated direct controller type builder generates an empty or minimal spec/status when mapping a resource from scratch, especially missing custom reference fields (such as `NetworkRef` and `SubnetworkRef`) and missing properties if they don't map automatically from Proto definitions without explicit Go type declarations. Also, standard greenfield templates default to `alpha` stability, while this is an existing `stable` resource.
- **Solution**: We hand-wrote the necessary fields (`defaultPort`, `description`, `location`, `networkEndpointType`, `networkRef`, `resourceID`, `subnetworkRef`) in `ComputeNetworkEndpointGroupSpec` and status properties (`selfLink`, `size`) in `ComputeNetworkEndpointGroupStatus` matching the existing CRD definition perfectly, ensuring proper validation tags like `// +kubebuilder:validation:Enum` and `// +required`. We also used `stability-level=stable` to keep metadata aligned with the CRD's existing stable status.
- **Impact**: Ensures that when running generation for existing resources being migrated or completed, the generated CRD schema does not experience breaking modifications or regression of fields.
