# ComputeForwardingRule Fuzzer Journal

## Observations & Learnings

- **Silent Exclusions via Incorrect Naming:** We found that `.ip_address` and `.ip_protocol` were incorrectly registered as Spec fields in the fuzzer. In the `pb.ForwardingRule` proto, the actual field descriptor names are `.I_p_address` and `.I_p_protocol`. Since they were registered with `.ip_address` and `.ip_protocol`, they did not match the descriptor names. Combined with `.I_p_address` and `.I_p_protocol` being listed in `Unimplemented_Internal`, these fields were completely ignored by fuzzing without raising any failures. Correcting the paths to `.I_p_address` and `.I_p_protocol` and removing them from `Unimplemented_Internal` enabled real fuzzing on these fields.
- **Reference URL Normalization via Filters:** The `.target` field is a string representation of reference URLs in the protobuf message. Because a randomly generated fuzzed string does not match standard GCP URL formats and contains no recognizable reference-type substrings (e.g. `targetHttpProxies`), the mapper's reference parsing returns an empty target, causing round-trip data loss. Using `f.FilterSpec` and `f.FilterStatus` to sanitize the fuzzed `.target` string to a valid, mockable URL allows the fuzzer to verify that the value is correctly passed through and mapped.
- **Filtering Nested Unmapped Fields:** The `serviceDirectoryRegistrations` field is fuzzed under `service_directory_registrations` but the nested field `service_directory_region` is not exposed in KRM. To prevent false round-trip failures, the filter function cleans up `service_directory_region` from all fuzzed registrations.
- **Standardizing Spec vs Status Fields:** The `.target` field exists in both KRM Spec and KRM Status. Listing it as both `f.SpecField` and `f.StatusField` causes it to be ignored entirely under both Spec and Status fuzzing runs. We solved this by listing `.target` solely as a `f.SpecField`, ensuring its mapping logic is fully fuzzed during the Spec fuzz test.
