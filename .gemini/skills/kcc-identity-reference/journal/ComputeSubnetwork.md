# ComputeSubnetwork Identity and Reference Journal

## Observations & Learnings

- **Type Mismatch Across Packages:** Since `ComputeSubnetworkRef` was moved from `apis/refs/v1beta1` to `apis/compute/v1beta1` to support identity-ref integration and standard `Normalize()` flow, several other APIs/KRM resources (such as `BatchJob`, `ColabRuntimeTemplate`, `CloudComposerEnvironment`, `ComputeNetworkAttachment`, `ContainerCluster`, `DataflowJob`, `DataflowFlexTemplateJob`, `ManagedKafkaCluster`, `MetastoreService`, `NetworkConnectivityRegionalEndpoint`, `NetworkConnectivityServiceConnectionPolicy`, `NotebookInstance`, `RunJob`, `TPUVirtualMachine`, and `WorkstationCluster`) that reference `ComputeSubnetworkRef` had to be updated to import and use the new package `computev1beta1.ComputeSubnetworkRef`.
- **Refactoring Mapper Files:** Multiple handwritten and autogenerated direct controller mapping files (under `pkg/controller/direct/`) also needed updates to use the new `computev1beta1.ComputeSubnetworkRef` type and resolve un-imported reference packages.
- **Normalization Instead of Resolver Helpers:** By implementing `refs.Ref` interface and standard `Normalize()` method on `ComputeSubnetworkRef`, we were able to completely eliminate the custom `ResolveComputeSubnetwork` helper in `computerefs.go` and local Direct controllers, making dependency resolution much cleaner, standard, and highly maintainable.
- **No Schema Changes:** The schema of `ComputeSubnetwork` has not been altered, complying strictly with safety protocols.
