# ComputeHTTPHealthCheck Journal Entry

## Findings & Learnings

- **Proto Message Name Overlap / Conflict**: In Google Compute API v1, there is a nested proto message `HTTPHealthCheck` under `HealthCheck`, but the legacy global resource is `/global/httpHealthChecks`. In the protobuf definitions, there is only a single `HTTPHealthCheck` message type, representing the structure of a health check. Because of this, when transitioning the legacy `ComputeHTTPHealthCheck` KCC resource, we map its kind `ComputeHTTPHealthCheck` to the `HTTPHealthCheck` proto message.
- **Strict Schema Compatibility vs Proto Representation**: The legacy `ComputeHTTPHealthCheck` CRD schema contains top-level properties like `checkIntervalSec`, `description`, `healthyThreshold`, `timeoutSec`, and `unhealthyThreshold`, which are not fields of the nested `HTTPHealthCheck` proto message (they are only present in the modern unified `HealthCheck` proto message). To maintain **strict schema compatibility** with the baseline CRD, we hand-coded the `ComputeHTTPHealthCheckSpec` and `ComputeHTTPHealthCheckStatus` structs inside `httphealthcheck_types.go` instead of relying entirely on auto-generated proto fields.
- **Spec / Status Labels and Fields**: The baseline CRD did not contain `spec.projectRef` or `spec.location` (which is standard for direct resources). Per `SKILL.md`, we omitted these fields to prevent breaking changes to the CRD schema. We successfully preserved the status fields `creationTimestamp` and `selfLink`, and maintained the original labels like `tf2crd=true` and `stability-level=stable`.
- **Custom Fuzzer Registration**: Registered a fuzzer `computeHTTPHealthCheckFuzzer()` using the nested `pb.HTTPHealthCheck` proto message type. Since `pb.HTTPHealthCheck` doesn't have the unmapped KRM fields (like `checkIntervalSec`), we only fuzzed `host`, `port`, and `request_path`, marking the remaining proto fields as `.Unimplemented_NotYetTriaged`.
