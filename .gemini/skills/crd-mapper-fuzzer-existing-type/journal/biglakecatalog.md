# BigLakeCatalog Journal

## Context
When transitioning the existing `BigLakeCatalog` resource (and its service `bigquerybiglake` which has both `v1alpha1` and `v1beta1`), we investigated the existing KRM types and generators configuration.

## Key Learnings & Resolutions
1. **Multi-version Mapper Handling**: 
   - `apis/bigquerybiglake/v1alpha1/generate.sh` was incorrectly executing a local, standalone `generate-mapper` invocation without `--multiversion`, which ended up stomping/colliding with the multi-version mapper generated by `apis/bigquerybiglake/v1beta1/generate.sh`.
   - We resolved this by removing the redundant `generate-mapper` step from `v1alpha1/generate.sh` entirely, in line with the `generate-sh-checker` skill which directs that mapper generation for multi-version services should only be invoked from one of the version's scripts (usually `v1beta1`) with the `--multiversion` flag.

2. **Skipped Output References**:
   - We added `--include-skipped-output` to the `generate-types` command in `apis/bigquerybiglake/v1alpha1/generate.sh` so that types previously skipped (due to existing non-generated Go types matching the proto tags) are generated but commented out. This provides a clear, valuable reference of what was omitted/skipped.

3. **Strict Schema Compatibility & CRD Manifests**:
   - Running `dev/tasks/diff-crds` on the generated CRDs outputted an empty diff, verifying that our direct types are strictly schema-compatible with the baseline CRDs.
